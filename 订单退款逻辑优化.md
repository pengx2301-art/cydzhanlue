# 订单退款逻辑优化

## 📋 需求说明

订单状态变更需要区分操作来源：
- **管理员手动操作**：点击按钮时，失败不自动退款
- **上游API回调**：自动操作时，失败自动退款

## ✅ 实现方案

### 1. 新增参数：`auto_refund`

在更新订单状态的API中增加 `auto_refund` 参数：

```javascript
PUT /api/orders/:id/status
{
  "status": 3,           // 订单状态
  "remark": "...",        // 备注
  "auto_refund": false   // 是否自动退款（默认false）
}
```

### 2. 余额变动逻辑

#### 消费订单（consume）

**管理员手动操作（auto_refund=false）：**
- 订单失败：不退款
- 订单退款：退款

**上游API回调（auto_refund=true）：**
- 订单失败：自动退款
- 订单退款：自动退款

#### 充值订单（recharge）

**无论是否手动操作：**
- 订单成功：增加余额
- 订单失败：不增加余额
- 订单退款：扣除余额（因为之前已经加过了）

## 🔧 API使用示例

### 管理员手动操作（前端）

```javascript
// 订单失败 - 不退款
await apiRequest('PUT', `/api/orders/${id}/status`, {
  status: 3,
  remark: '订单失败',
  auto_refund: false  // 不自动退款

// 订单退款 - 退款

  status: 4,

  auto_refund: false  // 虽然是退款状态，但管理员手动操作会退款
});
```

  auto_refund: true  // 自动退款
});

// 上游退款回调 - 自动退款
await apiRequest('PUT', `/api/orders/${id}/status`, {
  status: 4,

  auto_refund: true  // 自动退款

## 💡 核心逻辑说明

### 为什么需要区分操作来源？

1. **管理员手动失败**：
   - 可能是临时问题、待重试
   - 需要人工判断是否应该退款
   - 不自动退款，避免误操作

2. **上游API失败**：
   - 上游已经确认失败
   - 需要立即退款给用户
   - 自动退款，提升用户体验

### auto_refund 参数的作用

| 参数值 | 适用场景 | 消费订单失败 |
|--------|---------|-------------|
| `false` | 管理员手动操作 | 不退款 |
| `true` | 上游API回调 | 自动退款 |

**注意**：退款状态（status=4）总是退款，无论 `auto_refund` 值如何。

## 🔧 代码实现摘要

### 后端 API（server.js）

```javascript
// PUT /api/orders/:id/status
if (orderStatusMatch && req.method === 'PUT') {
  const { status, remark, auto_refund } = body;

  // 消费订单：根据 auto_refund 决定是否退款
  if (order.order_type === 'consume') {
    if (status === 4 && oldStatus !== 4) {
      // 退款状态：总是退还余额
      退还余额();
    } else if (status === 3 && oldStatus !== 3 && auto_refund === true) {
      // 失败状态 + auto_refund=true：退还余额
      退还余额();
    }
    // 失败状态 + auto_refund=false：不退款
  }
}
```
